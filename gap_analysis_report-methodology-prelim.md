This document reports the findings of a thorough cross-check between the research methodology described in [methodology-draft.pdf](file:///c:/Users/jw/OneDrive - Universiti Malaya/Sem_2 Study Material/WQF7023/repo/docs/methodology-draft.pdf) (Chapter 4) and the active implementation code in the repository.

## 1. Newly Identified Critical Gaps

Direct inspection of the repository files has revealed several significant discrepancies where the codebase deviates from the stated methodology or design specifications:

### Gap 1: Window Stride Subsampling (Stride = 7 vs. Stride = 1)
* **Methodology Requirement (Section 4.4)**: 
  * "Training samples are generated by applying a **one-day sliding window** over the training period... The window **advances one day at a time**, allowing all valid input-target pairs within the training period to be used for model training." (Page 32)
* **Active Code Implementation**:
  * In [configs/dataset.yaml](file:///c:/Users/jw/OneDrive - Universiti Malaya/Sem_2 Study Material/WQF7023/repo/configs/dataset.yaml#L7):
    ```yaml
    window_stride: 7          # Sliding window step size; 1 = fully overlapping, 7 = weekly stride
    ```
  * In [prelim.yaml](file:///c:/Users/jw/OneDrive - Universiti Malaya/Sem_2 Study Material/WQF7023/repo/configs/experiment/prelim.yaml#L3):
    ```yaml
    window_stride: 7
    ```
  * In [dataset.py](file:///c:/Users/jw/OneDrive - Universiti Malaya/Sem_2 Study Material/WQF7023/repo/data/dataset.py#L131):
    ```python
    stride = getattr(self.cfg.dataset, "window_stride", 1)
    if stride > 1:
        time_col = "time_idx_first_prediction"
        part_ds = part_ds.filter(lambda idx: idx[time_col] % stride == 0)
    ```
* **Impact**: **High**. Instead of sliding day-by-day (yielding a dense set of overlapping sequences), the codebase uses a weekly stride of 7 to filter the windows. While this reduces dataset size and speeds up training, it means the model only trains on 14% of the available chronological windows, directly violating the daily sliding-window requirement.

### Gap 2: Student Model Architecture — Omission of Future (Decoder) Inputs
* **Methodology Requirement (Section 4.6.3 & Table 4.4)**:
  * The student model must use the **shared input representation** defined in Section 4.3.2 (Table 4.1), which explicitly includes **Known Future Inputs** (SNAP days, price changes, holidays, calendar events).
* **Active Code Implementation**:
  * In the student forward pass in [student.py](file:///c:/Users/jw/OneDrive - Universiti Malaya/Sem_2 Study Material/WQF7023/repo/models/student.py#L63):
    ```python
    def forward(self, x):
        batch_size = x['encoder_cat'].shape[0]
        # Embed and concatenate encoder categoricals and continuous variables
        embedded = []
        for i, embed_layer in enumerate(self.embeddings):
            cat_tensor = x['encoder_cat'][:, :, i].long()
            ...
        embedded_tensor = torch.cat(embedded, dim=-1)
        x_full = torch.cat([embedded_tensor, x['encoder_cont']], dim=-1)
        
        x_proj = self.input_projector(x_full)
        enc_out = self.transformer_encoder(x_proj)
        
        # Flatten and project to predictions
        enc_flat = enc_out.reshape(batch_size, -1)
        preds = self.output_layer(enc_flat)
        return preds
    ```
* **Impact**: **High**. The student model completely ignores `x['decoder_cat']` and `x['decoder_cont']`. Unlike the TFT teacher (which ingests future SNAP days, selling prices, and events via its decoder structure), the student's Transformer encoder only processes historical observations. The future known variables are discarded, meaning the student is blind to upcoming calendar features and price changes.

### Gap 3: Hardcoded Embedding Dimensions (Violates Design Specifications)
* **Methodology Requirement**:
  * Principle 1 of [repository-design-specs.md](file:///c:/Users/jw/OneDrive - Universiti Malaya/Sem_2 Study Material/WQF7023/repo/docs/repository-design-specs.md#L30) states: "All experiment settings must come from configuration files... No model hyperparameters should be hardcoded inside Python source files."
* **Active Code Implementation**:
  * In [student.py](file:///c:/Users/jw/OneDrive - Universiti Malaya/Sem_2 Study Material/WQF7023/repo/models/student.py#L34-L39):
    ```python
    self.embeddings = nn.ModuleList([
        nn.Embedding(
            num_embeddings=len(training_dataset._categorical_encoders[col].classes_) + 1,
            embedding_dim=8  # Hardcoded dimension
        ) for col in self.cat_cols
    ])
    ```
* **Impact**: **Low to Moderate**. Sticking the embedding dimension value directly in the code breaks configuration flexibility and goes against the project's own design specifications.

## 2. Previously Documented Gaps

### Gap 4: Distillation Loss Mismatch (Huber vs. MSE)
* **Discrepancy**: The research description specifies **Huber Loss** for supervised learning (handling spikes) and **MSE Loss** for distillation (matching continuous teacher predictions).
* **Code Implementation**: In [student.py](file:///c:/Users/jw/OneDrive - Universiti Malaya/Sem_2 Study Material/WQF7023/repo/models/student.py#L110-L111), `nn.HuberLoss()` is used for both `loss_sup` and `loss_dist`.

### Gap 5: Missing Training Time Benchmarks
* **Discrepancy**: Table 4.6 in the PDF lists `training time` as a metric.
* **Code Implementation**: Elapsed training duration is not timed or stored in the experiment metadata files.

### Gap 6: MASE Denominator Seasonal Lag (28 vs. 7)
* **Discrepancy**: The MASE calculation in [evaluate_models.py](file:///c:/Users/jw/OneDrive - Universiti Malaya/Sem_2 Study Material/WQF7023/repo/scripts/evaluate_models.py#L189) uses a lag of 28 (`shift(28)`) to construct the baseline seasonal naive error. Standard retail MASE benchmarks for daily datasets with weekly seasonality typically use a lag of 7 (`shift(7)`).
